# 2. システム概要

## 2.1 システム名

Studio Book

## 2.2 システム種別

スタジオ時間貸し予約サービス

## 2.3 システム概要

Studio Bookは、ダンススタジオ、撮影スタジオ、音楽練習室、レンタルスペース等の時間貸し予約を想定したWebアプリケーションである。

一般ユーザーは、スタジオ検索、AI自然文検索、空き状況確認、予約、Stripe決済、予約履歴確認、レビュー投稿を行う。

スタジオ提供者であるホストは、所有スタジオ、営業時間、休館日、料金ルール、予約、レビュー、売上および統計を管理する。

管理者は、ユーザー、スタジオ、予約、アプリケーション設定、監査ログおよびAI検索ログを横断的に管理する。

## 2.4 開発背景

既存のJava / Spring Boot版Studio Bookを参考にしつつ、ASP.NET Core Web APIとNext.js App Routerによるフロントエンド・バックエンド分離構成として再設計・再実装する。

本システムは、以下を示すフルスタックポートフォリオとして構築する。

- ASP.NET Core Web APIによる業務API設計
- Next.jsによる複数ロール対応Web UI
- JWT認証・ロール別認可
- 予約・料金計算・決済・レビューを含む業務フロー
- Stripe、OpenAI、メール送信サービスとの外部連携
- MySQL・Entity Framework Coreによる永続化
- 管理ログ、AI検索ログ、PDF・CSV出力
- GitHub ActionsによるCI
- Azure Static Web AppsとHerokuによる分離デプロイ

# 3. システム目的

## 3.1 業務目的

1. 利用者が目的、地域、料金、人数等からスタジオを検索できること。
2. 利用者がスタジオの営業時間、休館日、料金ルール、レビューおよび空き状況を確認できること。
3. 利用者が希望日時を指定し、料金を確認した上でオンライン決済できること。
4. ホストが所有スタジオと予約状況を一元管理できること。
5. ホストが営業時間、休館日および料金ルールを設定できること。
6. ホストが売上と利用状況を確認できること。
7. 管理者がサービス全体のユーザー、スタジオ、予約および設定を管理できること。
8. 管理操作やAI検索利用状況を追跡できること。

## 3.2 技術的目的

1. フロントエンドとバックエンドを独立して開発・デプロイできる構成とする。
2. 認証・認可をAPI側で保証する。
3. Controller、Service、Entity、DTOの責務を分離する。
4. 外部サービス連携を専用ServiceまたはClientへ分離する。
5. MigrationによりDBスキーマを管理する。
6. 自動テストとCIにより品質を確認できること。
7. 環境変数・Secretsにより環境依存設定と秘密情報を管理する。

# 5. 利用者・ロール

## 5.1 利用者区分

| 利用者 | 概要 |
|---|---|
| 未認証ユーザー | トップ、規約、スタジオ検索・詳細、会員登録、ログインを利用する |
| 一般ユーザー | スタジオを検索・予約・決済し、利用後にレビューを投稿する |
| ホスト | 自身が所有するスタジオと予約・売上・レビューを管理する |
| 管理者 | システム全体のユーザー、スタジオ、予約、設定、ログを管理する |

## 5.2 ロール

| ロール | 説明 |
|---|---|
| GeneralUser | 一般利用者 |
| Host | スタジオ提供者 |
| Admin | 管理者 |

## 5.3 権限基本方針

1. 未認証ユーザーは公開画面のみ利用できる。
2. 一般ユーザーは自身の会員情報と予約情報のみ操作できる。
3. ホストは自身が所有するスタジオに関するデータのみ操作できる。
4. 管理者は全体管理機能を利用できる。
5. 画面表示だけでなく、API側で必ず認証・認可を行う。
6. 権限不足時はHTTP 403を返す。
7. 未認証時はHTTP 401を返す。

# 6. 業務概要

## 6.1 会員登録・認証フロー

```text
会員登録
  ↓
認証メール送信
  ↓
メール内リンクを開く
  ↓
メール認証完了
  ↓
ログイン可能
```

### 業務要件

1. 会員登録時に氏名、フリガナ、メールアドレス、住所、電話番号、利用区分、パスワードを入力する。
2. メールアドレスは重複登録できない。
3. 登録後、メール認証が完了するまで本登録扱いとしない。
4. 認証トークンには有効期限を設ける。
5. 認証トークンは一度のみ使用可能とする。
6. パスワードはハッシュ化して保存する。

## 6.2 パスワード再設定フロー

```text
再設定申請
  ↓
再設定メール送信
  ↓
メール内リンクを開く
  ↓
新しいパスワード入力
  ↓
パスワード更新
```

### 業務要件

1. 再設定トークンには有効期限を設ける。
2. 使用済みトークンは再利用できない。
3. パスワードは8文字以上を基本とする。
4. 新しいパスワードと確認用パスワードの一致を確認する。

## 6.3 スタジオ検索フロー

```text
トップ
  ↓
スタジオ一覧
  ├─ 通常検索
  └─ AI自然文検索
       ↓
スタジオ詳細
```

### 業務要件

1. スタジオ名、住所、地域、上限料金等で検索できる。
2. 検索結果はページング表示する。
3. スタジオ詳細には料金、定員、住所、説明、営業時間、料金ルール、レビューおよび空き状況を表示する。
4. AI検索では自然文を構造化された検索条件へ変換する。
5. AI検索結果には候補理由を表示する。
6. AI結果は補助情報であり、正確性を保証しない。

## 6.4 予約・決済フロー

```text
スタジオ詳細
  ↓
予約日時入力
  ↓
予約内容・料金確認
  ↓
Stripe Checkoutセッション作成
  ↓
Stripe決済
  ↓
Webhook受信
  ↓
予約状態更新
  ↓
予約履歴へ反映
```

### 業務要件

1. 予約はGeneralUserのみ実行できる。
2. 予約開始日時と終了日時を入力する。
3. 終了日時は開始日時より後でなければならない。
4. 営業時間外、休館期間、既存予約との重複を禁止する。
5. 料金は基本料金、時間帯料金、固定費、税、管理手数料等を考慮して計算する。
6. 料金確認後にStripe Checkoutへ遷移する。
7. 決済完了は画面遷移のみで確定せず、Stripe Webhookを基準に反映する。
8. 決済完了後は予約状態を`paid`へ更新する。
9. キャンセル時は予約状態を`canceled`へ更新する。
10. 過去予約の料金内訳は、料金ルール変更後も保持する。

## 6.5 レビューフロー

```text
利用済み予約
  ↓
レビュー投稿画面
  ├─ 評価入力
  ├─ コメント入力
  └─ AI文章補助
       ↓
ユーザー確認・修正
       ↓
投稿
```

### 業務要件

1. 対象スタジオを利用した一般ユーザーのみ投稿できる。
2. 評価は1～5とする。
3. コメントは必須とする。
4. AIレビュー補助は入力文を整えるのみとし、自動投稿しない。
5. ホストはレビューへ返信できる。
6. レビューは公開・非公開状態を管理できる。

## 6.6 ホスト管理フロー

1. ホストは自身の所有スタジオ一覧を確認できる。
2. 曜日別営業時間を登録・更新できる。
3. 臨時休館期間を登録・削除できる。
4. 曜日・時間帯別の倍率料金または固定費ルールを設定できる。
5. 自身のスタジオに関する予約一覧を確認できる。
6. 予約の承認・キャンセル等の状態変更を行える。
7. レビューへ返信し、公開状態を管理できる。
8. 売上一覧、売上明細、統計を確認できる。
9. 売上データをCSV・PDFで出力できる。

## 6.7 管理者管理フロー

1. 管理者は全ユーザーの一覧・詳細を確認できる。
2. 管理者は全スタジオの一覧・詳細を確認できる。
3. 管理者はスタジオを登録・編集できる。
4. 管理者は全予約を検索・確認できる。
5. 管理者は税率・管理手数料率等の設定を変更できる。
6. 管理者は全体統計を確認できる。
7. 管理者は監査ログを検索・確認できる。
8. 管理者はAI検索ログを検索・確認できる。

# 9. API要件

## 9.1 基本方針

1. フロントエンドとバックエンド間はHTTP/HTTPSおよびJSONで通信する。
2. APIはREST指向のエンドポイントを提供する。
3. 認証が必要なAPIはJWTを検証する。
4. 認可が必要なAPIはロールとデータ所有者を検証する。
5. 正常時は用途に応じた2xxステータスを返す。
6. 入力不正は400、未認証は401、権限不足は403、対象なしは404、競合は必要に応じ409、過剰利用は429、予期しない障害は500を返す。
7. エラーレスポンスは利用者向けメッセージを含める。

## 9.2 主要API分類

| 分類 | 例 |
|---|---|
| 認証 | `/api/auth/*` |
| 公開スタジオ | `/api/home`、`/api/rooms/*` |
| 予約 | `/api/reservations/*` |
| レビュー | `/api/rooms/{roomId}/reviews*` |
| AI | `/api/ai/*` |
| ホスト | `/api/host/*` |
| 管理者 | `/api/admin/*` |
| Stripe Webhook | Stripe Webhook受信用API |

詳細なAPI仕様はControllerおよび今後作成するAPI一覧・詳細設計を参照する。

| 会員 | Users、Roles、UserRoles |
| 認証 | VerificationTokens、PasswordResetTokens |
| スタジオ | Rooms、BusinessHours、Closures、PriceRules |
| 予約 | Reservations、ReservationChargeItems |
| レビュー | Reviews |
| 設定 | AppSettings |
| ログ | AuditLogs、AiSearchLogs |

## 10.2 データ基本要件

1. メールアドレスは一意とする。
2. ロール名は一意とする。
3. 同一ユーザーへの同一ロール重複付与を禁止する。
4. 同一スタジオの同一曜日営業時間は1件とする。
5. 同一名称・同一住所のスタジオ重複登録を禁止する。
6. 認証トークンと再設定トークンは一意とする。
7. 予約・レビューが存在するデータは履歴保全を優先する。
8. ユーザーは原則として物理削除せず、Enabledにより無効化する。
9. 予約確定時点の料金内訳を保存する。
10. 監査ログとAI検索ログは元データ削除後も保持できる構造とする。

詳細は `database-requirements.md` を参照する。

# 11. 認証・認可要件

## 11.1 認証方式

現行実装はJWT Bearer認証を採用する。

バックエンドは以下からJWTを取得できる。

- Authorizationヘッダー
- `auth_token` Cookie

フロントエンド現行実装では、JWTをクライアント側で保持しAuthorizationヘッダーへ設定する構成が含まれる。

## 11.2 要件

1. Issuerを検証する。
2. Audienceを検証する。
3. 署名鍵を検証する。
4. 有効期限を検証する。
5. ClockSkewは0とし、有効期限を厳密に判定する。
6. JWT署名鍵をソースコードへ保存しない。
7. パスワードを平文保存しない。
8. 本番通信はHTTPSを使用する。
9. Cookieを正式利用する場合、Secure、HttpOnly、SameSiteを適切に設定する。
10. JWTの正式な保存方式は今後確定する。

# 12. 外部サービス連携要件

## 12.1 Stripe

1. Stripe Checkoutによりテストモード決済を行う。
2. SecretKey、PublishableKey、WebhookSecretを環境設定から取得する。
3. カード情報はシステム内へ保存しない。
4. 決済完了はWebhook署名検証後に反映する。
5. Webhook再送に対し冪等に処理する。
6. Stripe障害時は予約を決済済みにしない。
7. SuccessUrlとCancelUrlを環境ごとに設定する。

## 12.2 OpenAI

1. OpenAI APIをAIスタジオ検索とAIレビュー補助に利用する。
2. APIキーとモデル名を環境設定から取得する。
3. AI検索は同一IPアドレスにつき1分5回までとする。
4. 上限超過時はHTTP 429を返す。
5. AI検索の成功・失敗・結果件数を記録する。
6. OpenAI障害時も通常検索や予約機能は利用可能とする。
7. AI出力を確定情報として扱わない。

## 12.3 メール送信

1. 会員認証メールを送信する。
2. パスワード再設定メールを送信する。
3. 開発・検証環境ではMailtrap等を使用できる。
4. メール送信設定は環境変数・Secretsで管理する。
5. トークンには有効期限を設定する。

## 12.4 PDF・CSV

1. PDF生成にはQuestPDFを使用する。
2. 日本語フォントとしてNoto Sans JPを使用する。
3. 日本語が文字化けしないこと。
4. 売上情報をCSV出力できること。
5. CSVの区切り、改行、ダブルクォートを適切に処理する。

# 20. 現行実装との差異・要確認事項

## 20.1 JWT保存方式

READMEにはHttpOnly Cookie認証と記載されている一方、フロントエンドにはlocalStorageのトークンをAuthorizationヘッダーへ設定する実装が存在する。

正式方式を以下のいずれかに統一する必要がある。

- HttpOnly Cookie
- Authorization Bearer
- Bearerを主方式としCookieを互換方式とする

## 20.2 Swagger認証

現行の `AddSwaggerGen()` だけでは、Swagger UI上のBearer Security Definition設定を確認できない。

Swaggerから認証APIを試験する場合の方法を確定する必要がある。

## 20.3 .NET・EF Coreバージョン

READMEではASP.NET Core 8と記載され、ModelSnapshotではEntity Framework Core 9.0.1が確認されている。

ターゲットフレームワーク、EF Core、MySQL Providerの正式な組合せを確認する。

## 20.4 予約状態

現行の主な予約状態は以下である。

- `booked`
- `paid`
- `canceled`

`booked`が決済待ち、予約受付済み、仮予約のいずれを意味するかを明文化する必要がある。

## 20.5 レビューと予約の関連

ReviewsテーブルにReservationIdがないため、DB上で1予約1レビューを直接保証できない。

現行のService側判定を確認し、必要に応じReservationId追加を検討する。

## 20.6 予約重複制御

DB制約では予約時間帯の重複を防げない。

Service側のトランザクション・排他制御の実装状況を確認する。

## 20.7 曜日番号

BusinessHoursとPriceRulesで曜日番号体系が異なる。

- BusinessHours：月=1～日=7
- PriceRules：日=0～土=6、NULL=全曜日

共通変換処理と設計書記載を徹底する。

## 20.8 URL表記

一部画面・リンクで `/auth/login` と `/login` が混在している。

正式URLを `/auth/login` 等に統一する。

## 20.9 文字化け

Program.cs内の一部コメントおよびレート制限メッセージに文字化けが見られるため、UTF-8へ修正する。
