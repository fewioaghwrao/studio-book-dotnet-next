# Studio Book 非機能要件書

## 1. 文書概要

本書は、Studio Book の現行実装から確認できる非機能要件と、今後の運用・公開を想定した暫定目標値を整理したものである。

本書では、各要件を以下の区分で記載する。

| 区分 | 意味 |
|---|---|
| 現行実装 | ソースコード、設定クラス、ミドルウェア構成などから確認できる内容 |
| 暫定目標 | 現時点では未実装または未計測だが、ポートフォリオ・小規模公開サービスとして妥当と考えられる目標 |
| 要確認 | 環境、運用方針、外部サービス契約などにより確定が必要な内容 |

---

## 2. 対象システム

| 項目 | 内容 |
|---|---|
| システム名 | Studio Book |
| システム種別 | スタジオ予約Webアプリケーション |
| フロントエンド | Next.js |
| バックエンド | ASP.NET Core Web API |
| DB | MySQL |
| 認証 | JWT Bearer |
| 外部サービス | Stripe、OpenAI、メール送信サービス |
| PDF生成 | QuestPDF |
| 想定利用者 | 一般ユーザー、ホスト、管理者 |

# 3. 性能要件

## 3.1 API応答時間

| 対象 | 現行実装 | 暫定目標 |
|---|---|---|
| 一覧・詳細API | 個別のタイムアウト・SLA設定は未確認 | 通常時95%以上を3秒以内 |
| ログイン・会員登録 | 個別目標値なし | 通常時3秒以内 |
| 管理画面一覧 | ページング実装あり | 10件単位表示で3秒以内 |
| AIスタジオ検索 | OpenAI APIに依存 | 30秒以内 |
| AIレビュー補助 | OpenAI APIに依存 | 30秒以内 |
| CSV出力 | 実装あり | 10秒以内 |
| PDF出力 | QuestPDFで生成 | 10秒以内 |
| Stripe決済開始 | Stripe APIに依存 | 10秒以内にCheckoutへ遷移 |

### 要件

1. 通常の一覧・詳細・更新APIは、通常負荷時に3秒以内で応答することを目標とする。
2. AI機能、PDF出力、外部決済など外部処理を含む機能は30秒以内を上限目標とする。
3. 一覧画面は原則としてページングを使用し、一度に大量データを返却しない。
4. APIの応答時間は今後ログまたはAPMで計測可能にする。

---

## 3.2 同時利用・処理能力

| 項目 | 現行実装 | 暫定目標 |
|---|---|---|
| 想定同時利用者数 | 未定義 | 20ユーザー程度 |
| 通常API同時リクエスト | 未計測 | 50リクエスト程度に耐える |
| AI検索 | IP単位で制限あり | 同一IPで1分5回 |
| 待ち行列 | AI検索はQueueLimit=0 | AI検索は待ち行列なし |

### 要件

1. 小規模公開デモとして、同時20ユーザー程度を想定する。
2. AI検索は同一IPアドレスにつき1分間に5回までとする。
3. AI検索の上限超過時は待ち行列へ入れず、HTTP 429を返す。
4. 通常APIについても将来的に負荷試験を実施し、必要に応じて追加のレート制限を検討する。

|---|---|---|

| APIルート確認 | `/` で案内を返却 | 死活監視に利用可能 |
| 稼働率 | 未定義 | 月間99.0%以上 |

### 要件

2. `/health` を監視対象とする。
3. 計画メンテナンスは可能な限り事前に告知する。
4. AI・Stripe等の外部サービス障害時も、通常のスタジオ検索や閲覧機能は継続可能とする。

## 4.2 障害復旧目標

| 項目 | 現行実装 | 暫定目標 |
|---|---|---|
| RTO | 未定義 | 4時間以内 |
| RPO | 未定義 | 24時間以内 |
| DB初期化失敗 | 起動失敗として扱う | 原因確認後に再デプロイまたは復旧 |
| 外部API障害 | 個別機能エラー | システム全体停止を避ける |

### 要件

1. 重大障害時のRTOは4時間以内を暫定目標とする。
2. RPOは24時間以内を暫定目標とする。
3. DBマイグレーションまたはSeed処理失敗時はアプリ起動を失敗させ、ログへ原因を記録する。
4. 外部サービス障害時は対象機能のみを停止し、他機能への影響を最小化する。

# 5. 信頼性要件

## 5.1 データ整合性

### 現行実装

- EF Coreによる外部キー制約を使用する。
- ReservationsとReviewsの削除にはRestrictが設定されている。
- ReservationChargeItemsはReservation削除時にCascadeされる。
- DBマイグレーションは環境変数で起動時実行を制御できる。

### 暫定目標

1. 予約確定、料金明細登録、決済結果反映はトランザクション内で処理する。
2. Stripe Webhookは同一イベントが再送されても二重更新しないよう冪等に処理する。
3. 同一スタジオ・同一時間帯の二重予約を防止する。
4. 料金ルール変更後も過去予約の料金明細が変わらないよう、予約確定時点の明細を保存する。
5. 管理設定変更と監査ログ記録は可能な限り同一トランザクションで処理する。

---

## 5.2 外部サービス障害

| サービス | 要件 |
|---|---|
| Stripe | 障害時は予約を決済完了扱いにしない |
| OpenAI | 障害時も通常検索・予約機能を継続できる |
| メール | 送信失敗時は再送またはエラー通知を可能にする |
| MySQL | 接続不可時は起動または対象処理を失敗させる |

# 6. セキュリティ要件

## 6.1 認証

### 現行実装

- JWT Bearer認証を使用する。
- Issuer、Audience、署名鍵、有効期限を検証する。
- `ClockSkew = 0` とし、期限切れを厳密に判定する。
- Authorizationヘッダーに加えて `auth_token` CookieからもJWTを取得可能とする。
- パスワードハッシュ化にはASP.NET Core IdentityのPasswordHasherを使用する。

### 暫定目標

1. 本番通信はHTTPS必須とする。
2. JWT署名鍵は十分な長さとランダム性を持たせる。
3. JWT署名鍵をソースコードへ記述しない。
4. アクセストークン有効期限は30～60分程度を暫定値とする。
5. 認証失敗時は401、権限不足時は403を返す。
6. 無効化されたユーザーはログイン不可とする。
7. Cookieを正式利用する場合はSecure、HttpOnly、SameSite属性を適切に設定する。

---

## 6.2 認可

| ロール | 主な権限 |
|---|---|
| GeneralUser | スタジオ予約、自身の予約閲覧、レビュー投稿 |
| Host | 所有スタジオ、営業時間、休館日、料金、予約、レビュー、売上管理 |
| Admin | 会員、スタジオ、予約、設定、監査ログ、AI検索ログ管理 |

### 要件

1. 管理者APIはAdminロールに限定する。
2. ホストAPIはHostロールに限定する。
3. ホストは自身が所有するスタジオのみ更新可能とする。
4. 一般ユーザーは自身の予約のみ閲覧・操作可能とする。
5. 画面側の表示制御だけでなく、必ずAPI側でも認可する。

---

## 6.3 秘密情報管理

### 対象

- DB接続文字列
- JWT署名鍵
- Stripe SecretKey
- Stripe WebhookSecret
- OpenAI API Key
- メール送信用認証情報

### 要件

1. 秘密情報は環境変数またはホスティングサービスのSecretsで管理する。
2. リポジトリへコミットしない。
3. ログへ出力しない。
4. 開発・検証・本番で値を分離する。
5. 漏えい時に速やかにローテーションできること。

---

## 6.4 入力・出力セキュリティ

1. API入力値は型、必須、長さ、範囲を検証する。
2. AI検索文は最大200文字を基本とし、バックエンド側でも上限を検証する。
3. レビュー本文は最大1000～2000文字の範囲で統一する。
4. HTMLとして表示する値は適切にエスケープする。
5. SQLはEF Core経由で実行し、文字列連結によるSQL生成を避ける。
6. PDF、CSV出力時に個人情報の過剰出力を避ける。
7. CSVインジェクション対策を必要に応じて実施する。

---

# 7. 通信・CORS要件

## 7.1 現行実装

- `FrontendPolicy` を使用する。
- 許可オリジンは `Cors:AllowedOrigins` から取得する。
- 未設定時は `http://localhost:3000` を許可する。
- 任意のHTTPヘッダーとHTTPメソッドを許可する。
- Credentialを許可する。
- HTTPSリダイレクトを有効にする。

## 7.2 要件

1. 本番環境では許可オリジンを明示的に限定する。
2. `AllowCredentials` 利用時はワイルドカードオリジンを使用しない。
3. 開発・本番で許可オリジンを分離する。
4. API通信は原則HTTPSとする。
5. 不要なHTTPメソッド・ヘッダーの制限は将来的に検討する。

---

# 8. レート制限・不正利用対策

## 8.1 現行実装

| 項目 | 内容 |
|---|---|
| ポリシー名 | AiSearchPolicy |
| 対象 | AIスタジオ検索 |
| 識別単位 | IPアドレス |
| 方式 | Fixed Window |
| 上限 | 1分5回 |
| 待ち行列 | なし |
| 超過レスポンス | HTTP 429 |
| 超過ログ | AiSearchLogsへ記録 |

## 8.2 要件

1. AI検索の過剰利用を防ぐ。
2. レート制限拒否時も検索文、IP、ユーザーID、失敗理由を可能な範囲で記録する。
3. IPアドレス取得時はリバースプロキシ配下のForwarded Headers設定を考慮する。
4. ログイン、会員登録、パスワード再設定にも将来的なレート制限を検討する。
5. レート制限値は環境設定化を検討する。

# 9. ログ・監査要件

## 9.1 ログ種別

| 種別 | 現行実装 | 要件 |
|---|---|---|
| アプリケーションログ | 標準ILoggerを利用 | 起動、例外、重要処理を記録 |
| DB初期化ログ | Migration・Seed開始／完了／失敗を記録 | 本番障害調査に利用 |
| 監査ログ | AuditLogsテーブル | 管理操作・設定変更等を記録 |
| AI検索ログ | AiSearchLogsテーブル | 検索文、IP、ユーザー、成否、件数、エラーを記録 |
| レート制限ログ | AiSearchLogsへ記録 | 429発生状況を追跡 |
| 認証ログ | 明示実装は要確認 | 失敗回数や不正利用を必要に応じて記録 |

## 9.2 暫定保持期間

| ログ | 暫定保持期間 |
|---|---|
| アプリケーションログ | 30日 |
| 監査ログ | 1年 |
| AI検索ログ | 90日 |
| 認証失敗ログ | 90日 |

### 要件

1. パスワード、JWT、秘密鍵、カード情報をログへ出力しない。
2. 監査ログは原則として更新・削除不可とする。
3. 個人情報を含むログへのアクセスはAdminに限定する。
4. ログ保存期間経過後の削除または匿名化を検討する。
5. 本番では標準出力ログをホスティング基盤で収集する。

# 11. DB運用要件

## 11.1 現行実装

- MySQLを使用する。
- EF Core Migrationを使用する。
- `ENABLE_DB_MIGRATION` により起動時Migrationを制御する。
- `ENABLE_DB_SEED` により起動時Seedを制御する。
- 初期化失敗時はログ記録後に例外を再送出し、起動を失敗させる。

## 11.2 要件

1. 本番環境でSeedは原則無効とする。
2. 本番Migrationは事前バックアップ後に実行する。
3. Migration実行権限と通常アプリ実行権限の分離を将来的に検討する。
4. Migration失敗時は自動的に処理を継続しない。
5. 本番反映前に検証環境でMigrationを確認する。
6. 大規模データ変更を含むMigrationはメンテナンス時間帯に実施する。

# 13. 外部サービス連携要件

## 13.1 Stripe

### 現行実装

- SecretKey、PublishableKey、WebhookSecret、SuccessUrl、CancelUrlを設定可能。
- SecretKeyが設定されている場合にStripe SDKを初期化する。
- Checkout、Webhook、決済完了処理用サービスが存在する。

### 要件

1. 本番ではWebhook署名検証を必須とする。
2. 同一Webhookの再送に対して冪等であること。
3. Stripe障害時は予約をpaidにしない。
4. SecretKeyとWebhookSecretは秘密情報として管理する。
5. カード番号をアプリケーションまたはDBへ保存しない。
6. Stripe APIタイムアウトは暫定30秒以内とする。
7. 決済失敗時は利用者へ再試行可能なメッセージを返す。

---

## 13.2 OpenAI

### 現行実装

- OpenAI API Keyとモデル名を設定可能。
- 既定モデルは `gpt-5.4-mini`。
- スタジオ検索とレビュー補助でHttpClientを利用する。
- AI検索ログを記録する。
- AIスタジオ検索にレート制限を適用する。

### 要件

1. OpenAI API障害時も通常検索・予約機能を継続可能とする。
2. タイムアウトは暫定30秒以内とする。
3. APIキーを秘密情報として管理する。
4. AI出力は候補・補助情報として扱い、正確性を保証しない。
5. AI検索文の保存についてプライバシーポリシーへ明記する。
6. 不適切な入力・出力への対策を必要に応じて追加する。
7. 利用料金・トークン使用量の監視を検討する。

---

## 13.3 メール送信

### 現行実装

- 会員認証メール、パスワード再設定メール用のサービスが存在する。

### 暫定要件

1. メール送信失敗時に処理結果を利用者へ通知する。
2. 認証・再設定トークンには有効期限を設定する。
3. 同一トークンの再利用を禁止する。
4. メール送信APIのタイムアウトは暫定15秒以内とする。
5. 送信失敗時の再送方法を将来的に整備する。

# 14. PDF・CSV要件

## 14.1 PDF

### 現行実装

- QuestPDF Communityを使用する。
- Noto Sans JP Regular/Boldを登録する。
- フォントファイルが存在する場合のみ登録する。

### 要件

1. 日本語が文字化けせず出力されること。
2. 本番配置物に必要なフォントを含めること。
3. フォントが存在しない場合はログ出力または起動時検証を行うことを検討する。
4. PDF生成失敗時は適切なHTTPエラーを返す。
5. PDF生成時間は通常10秒以内を目標とする。
6. PDFに秘密情報や不要な個人情報を含めない。

## 14.2 CSV

1. UTF-8または仕様で定めた文字コードで出力する。
2. Excel等で開いた際の文字化けを考慮する。
3. カンマ、改行、ダブルクォートを適切にエスケープする。
4. 数式として解釈される値へのCSVインジェクション対策を検討する。
5. CSV出力時間は通常10秒以内を目標とする。

# 16. 保守性・拡張性要件

## 16.1 現行実装

- Controller、Service、Service Interface、Settings、Seederを分離している。
- DIコンテナへ各Serviceを登録している。
- 外部APIクライアントはHttpClient Factoryを使用している。
- 機能ごとにServiceを分割している。
- 環境変数でSwagger、Migration、Seedを制御できる。

## 16.2 暫定目標

1. ControllerにはHTTP入出力と認可判断を中心に置き、業務ロジックはServiceへ分離する。
2. 外部サービス連携は専用ClientまたはServiceへ分離する。
3. 共通設定はOptionsパターンで管理する。
4. 単体テスト可能な設計を維持する。
5. 公開APIの破壊的変更を避ける。
6. 新規機能追加時に既存ロール・既存予約処理へ影響を与えない構造とする。
7. 警告や文字化けしたソースコメント・メッセージを解消する。

# 17. テスト要件

## 17.1 暫定目標

| テスト種別 | 対象 |
|---|---|
| 単体テスト | 料金計算、予約重複、権限判定、トークン判定 |
| 統合テスト | Controller、認証、DB、主要API |
| E2Eテスト | 会員登録、ログイン、検索、予約、決済後反映 |
| 外部連携テスト | Stripe Webhook、OpenAI異常系、メール送信 |
| 負荷テスト | 一覧API、予約確認、AI検索レート制限 |
| PDF・CSVテスト | 日本語、金額、文字化け、明細整合性 |

### 品質目標

1. 主要業務ロジックの単体テストを整備する。

# 20. 障害対応要件

1. 障害発生時は、発生日時、対象機能、HTTPステータス、例外概要を確認する。
2. 秘密情報を含まない範囲で原因調査用ログを残す。
3. DB初期化失敗時はMigration・接続文字列・権限を確認する。
4. Stripe障害時はWebhook履歴と予約状態の不整合を確認する。
5. OpenAI障害時は通常検索へ誘導できるようにする。
6. 重大障害は4時間以内の復旧を暫定目標とする。
7. 復旧後はデータ整合性を確認する。
8. 必要に応じて原因と再発防止策を記録する。
